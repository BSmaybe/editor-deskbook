# Система бронирования рабочих мест

## Быстрый старт

### Требования
- Docker и Docker Compose v2
- `jq` и `curl` (для запуска smoke-тестов)

### 1. Клонировать и настроить окружение
```bash
git clone <repo-url>
cd DO-main
cp .env.example .env
# Отредактируйте .env при необходимости (по умолчанию все значения готовы к локальной разработке)
```

Для production:
- `APP_ENV=production`
- задайте сильный `SECRET_KEY` (минимум 32 символа)
- укажите явные `ALLOWED_ORIGINS` (не `*`)

Для публичного ознакомления:
- включите `DEMO_MODE=true`, чтобы на лендинге работала кнопка «Попробовать демо».

### 2. Запустить все сервисы
```bash
docker compose up --build -d
```

| Сервис | URL |
|--------|-----|
| Клиентский интерфейс | http://localhost:5173 |
| Административная панель | http://localhost:5174 |
| React admin preview | http://localhost:5175 |
| REST API | http://localhost:8000 |
| Swagger UI | http://localhost:8000/docs |

### Маршрутизация на лэндинге (User/Admin/Demo)
- Лэндинг на `http://localhost:5173` работает как точка входа с тремя сценариями: пользователь, админ, демо.
- Авто-редирект сессий:
  - если есть валидный `admin_token`, переход в админку;
  - иначе если есть валидный `user_token`, переход в клиентское приложение;
  - просроченные токены очищаются и остается экран выбора режима.
- Кнопка «Войти как админ» использует:
  1. `window.__DESKBOOK_ADMIN_URL` (если задана);
  2. fallback на текущий хост с портом `5174`.

Если админка опубликована на отдельном домене, можно задать URL перед основным скриптом лэндинга в `frontend/client/index.html`:

```html
<script>
  window.__DESKBOOK_ADMIN_URL = "https://admin.example.com/";
</script>
```

### Hybrid SVG/XML map editor
- Основной формат редактирования карты: `layout_json` в ревизии этажа.
- При публикации backend генерирует `semantic_svg`: валидный SVG/XML с `<defs>/<symbol>`, иерархией `building -> storey -> zone -> workplace` и `data-*` атрибутами.
- Опубликованный SVG доступен как `GET /floors/{floor_id}/layout/published.svg` и возвращает `image/svg+xml` для внешних потребителей карты.
- Админский editor поддерживает базовые symbols: `desk-short`, `desk-long`, `chair`, `meeting-table`; source of truth остается JSON, SVG является опубликованным артефактом.
- Миграция на Go находится в `backend-go`: standalone/HTTP renderer `layout_json -> semantic SVG/HTML`, CRUD компонентов и layout draft/publish/export API с тем же контрактом.
- Первый React shell для editor-only админки находится в `frontend/admin-react`; он подключен отдельным compose-сервисом `admin-react` и не заменяет текущую статическую админку.
- Go editor API запускается вместе со стеком: `docker compose up -d --build renderer-go`.
- Чтобы FastAPI начал использовать Go renderer при публикации карт, задайте `GO_RENDERER_URL=http://renderer-go:8080` в `.env`; при недоступности сервиса backend откатится на Python renderer, если `GO_RENDERER_REQUIRED=false`.
- План миграции на Go backend + React editor-only UI: [docs/GO_REACT_EDITOR_MIGRATION.md](docs/GO_REACT_EDITOR_MIGRATION.md).

### 3. Проверить работоспособность (smoke-тест)
```bash
# Убедитесь, что ADMIN_REGISTER_SECRET задан в .env, если нужна регистрация администратора
bash tests/smoke_test.sh
bash tests/go_renderer_contract.sh
bash tests/go_components_contract.sh
bash tests/go_layout_contract.sh
```

Smoke-тест последовательно:
1. Ожидает готовности backend (`/health`)
2. Регистрирует тестового администратора и обычного пользователя
3. Входит в систему и получает JWT-токены
4. Создаёт офис, этаж и рабочее место через API администратора
5. Создаёт бронирование от имени пользователя
6. Проверяет защиту от двойного бронирования (ожидает HTTP 409)
7. Проверяет RBAC (неавторизованный запрос → 401, обычный пользователь на `/analytics` → 403)

### 4. Остановить сервисы
```bash
docker compose down
# Для полного удаления данных (включая базу данных):
docker compose down -v
```

## Цель проекта
Создать систему бронирования рабочих мест для офисов компании в рамках инициативы «цифровизации офиса». Решение ориентировано на работу внутри корпоративной инфраструктуры, использует open-source стек и поддерживает интеграцию с корпоративной аутентификацией.

## Основные функции

### 1. Поддержка нескольких офисов и этажей
- Три офисные локации, в том числе главный офис с двумя этажами.
- На каждом этаже используется план (например, PNG), где отображаются рабочие места и их статус.
- Небронирируемые зоны (кафе, лаунж) отображаются на плане для ориентации, но не доступны для брони.

### 2. Закрепленные и гибкие рабочие места
- **Закрепленные места**: назначаются конкретным сотрудникам или отделам и всегда считаются занятыми.
- **Гибкие места (hot desks)**: доступны для бронирования всеми сотрудниками.
- Администратор управляет статусом места (закрепленное/гибкое) через панель управления.

### 3. Правила бронирования
- **Проверка доступности** в реальном времени с защитой от двойного бронирования.
- **Гибкие слоты**: бронирование на весь день или на определенные часы.
- **Бронирование день-в-день или заранее** с ограничениями, задаваемыми политикой компании.
- **Повторяемые брони** (опционально), например, каждую неделю.
- **Подтверждение брони** через уведомления (email/Teams) и отображение в личном расписании.

### 4. QR-коды и подтверждение присутствия
- На каждом рабочем месте размещен уникальный QR-код.
- **Check-in**: сканирование QR-кода подтверждает прибытие и активирует бронь.
- **Мгновенное бронирование**: при отсутствии брони пользователь может отсканировать QR-код свободного места и сразу его занять.
- **Anti-no-show**: автоматическая отмена брони, если не было check-in в течение заданного времени (например, 15–30 минут).

### 5. Аутентификация и доступ
- Интеграция с корпоративным **SSO/LDAP** (например, Azure AD).
- Роли: **пользователь** (бронирование) и **администратор** (управление конфигурацией).
- Опционально: ограничение доступа по группам или подразделениям.

### 6. Интеграции (Microsoft Teams и другие сервисы)
- **Уведомления и напоминания** через Teams (подтверждение брони, напоминания, запрос на check-in) с возможностью параллельной отправки по email.
- **Бронирование через Teams** (опционально): приложение/бот для поиска и бронирования мест без выхода из Teams.
- **Интеграция с календарями** Outlook/Office 365: автоматическое добавление брони в календарь пользователя и экспорт в формате .ics.
- **Другие интеграции** (перспективно): контроль доступа, инфопанели офиса, Power BI для аналитики.

### 7. Административная панель
- Управление офисами и этажами, загрузка планов, интерактивная расстановка столов.
- Настройка рабочих мест: статус, тип, закрепление за сотрудниками/отделами, временное отключение.
- Политики бронирования: правила слотов, лимиты, таймаут no-show, ограничения по ролям/группам.
- Мониторинг броней и ручное администрирование (отмена/перенос).
- Аналитика занятости и отчеты, экспорт в BI-инструменты.
- Управление ролями, списком администраторов, настройками интеграций.

## Архитектура и технические решения

### Технологический стек
- **Frontend**: веб-интерфейс (SPA) на React/Vue/Angular с адаптивным дизайном для мобильных устройств.
- **Backend/API**: серверная логика (Node.js/Python/Java) с REST/GraphQL API.
- **Database**: PostgreSQL (или MariaDB) для хранения офисов, этажей, столов, пользователей и броней.
- **Интеграции**: модули для SSO/LDAP, Microsoft Graph API для Teams/Outlook, SMTP для email.
- **QR-коды**: генерация и хранение URL-меток для чек-ина и мгновенного бронирования.

### Деплой и инфраструктура
- Приоритет — **on-premises** развертывание в корпоративной сети.
- Контейнеризация (Docker/Kubernetes) для удобства обновлений и масштабирования.
- Доступ к LDAP/AD и, при необходимости, к внешним API Microsoft через прокси/шлюзы.

### Безопасность и права
- Аутентификация через корпоративный SSO без хранения паролей.
- Авторизация на уровне API: пользователь управляет только своими бронями, администратор — всем.
- HTTPS для всего трафика; аудит действий (логирование бронирований/отмен).

### Производительность и масштабируемость
- Кэширование статичных данных (планы этажей, список мест) на клиенте.
- Индексация по времени/месту в БД для быстрых проверок доступности.
- Горизонтальное масштабирование backend и вертикальное/реплицируемое масштабирование БД.
- Защита от конкурентных бронирований через транзакции/блокировки.

### UX/UI
- Интуитивная карта этажей со статусами мест.
- Быстрое бронирование в несколько кликов.
- Адаптивный дизайн для мобильных устройств.
- Потенциально: поиск коллег/зон, многоязычность.

## Минимальные сущности данных
- **Office**: офис, адрес, список этажей.
- **Floor**: этаж, план, список мест.
- **Desk**: рабочее место, статус (закрепленное/гибкое), зона.
- **Reservation**: бронь (пользователь, время, статус).
- **User**: учетная запись, роль.
- **Policy**: правила бронирования (лимиты, таймауты, слоты).
- **Integration**: настройки интеграций (Teams, email, календарь).
