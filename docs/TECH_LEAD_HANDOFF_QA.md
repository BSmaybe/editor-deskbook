# Пакет вопросов техлида для передачи DeskBook

Документ для встречи передачи внешней команде в формате: **вопросы + ожидаемые ответы**.

Цель: быстро проверить архитектуру, риски, эксплуатацию, безопасность и готовность к поддержке без текущих разработчиков.

## 1) Вопросы техлида и ожидаемые ответы

1. **Как поднимается проект локально и в стендах?**  
   Ожидаемый ответ: `docker compose up -d --build`; сервисы `postgres/backend/client/admin`; готовность backend через `/health`.

2. **Где точки входа системы?**  
   Ожидаемый ответ:  
   - Client: `http://localhost:5173`  
   - Admin: `http://localhost:5174`  
   - API: `http://localhost:8000`  
   - Swagger: `http://localhost:8000/docs`

3. **Как устроены роли и авторизация?**  
   Ожидаемый ответ: JWT с claim `role=user|admin`; админские операции защищены `require_admin` на backend.

4. **Как работает роутинг входа между User/Admin/Demo?**  
   Ожидаемый ответ: лэндинг проверяет `admin_token` с приоритетом над `user_token`; просроченные токены очищаются; переходы выполняются в текущей вкладке.

5. **Как ограничен demo-режим?**  
   Ожидаемый ответ: вход через `/auth/demo`, сессия только `role=user`; demo-пользователь ограничен по опасным мутациям на backend.

6. **Что по текущим security-рискам?**  
   Ожидаемый ответ: критичные XSS из review закрыты; redirect guard в client app не продолжает bootstrap после редиректа (см. `frontend/client/app.js`, guard и ранний выход из `init`).

7. **Как закрыт риск двойного бронирования?**  
   Ожидаемый ответ: пересечение слотов дает `HTTP 409`; сценарий покрыт smoke-тестом.

8. **Какие ограничения от brute force и спама?**  
   Ожидаемый ответ: на auth/demo применен `_rate_limit`; нужно показать текущие пороги и кто мониторит превышения.

9. **Как управляются конфиги и секреты?**  
   Ожидаемый ответ: `.env`; для production обязателен сильный `SECRET_KEY` (минимум 32 символа) и явные `ALLOWED_ORIGINS`.

10. **Как устроены миграции схемы БД и откат?**  
    Ожидаемый ответ: команда передачи фиксирует процесс миграций/rollback и владельца операций как отдельный handoff item (это организационно обязательный пункт на встрече).

11. **Какой минимальный набор тестов обязателен перед релизом?**  
    Ожидаемый ответ: `bash tests/smoke_test.sh` как release gate: auth, RBAC, создание сущностей, booking conflict.

12. **Что с наблюдаемостью?**  
    Ожидаемый ответ: показать где смотреть логи контейнеров (`docker compose logs ...`), какие алерты обязательны для `401/403/5xx`, кто on-call.

13. **Какой SLA/поддержка после передачи?**  
    Ожидаемый ответ: закреплены владелец инцидентов, окно реакции, канал эскалации, критерии критического простоя.

14. **Какие известные техдолги и что не входит в MVP?**  
    Ожидаемый ответ: сверка с roadmap (Teams/Outlook, BI, расширенная аналитика и др. будущие фазы).

15. **Какие доступы нужно передать внешней команде?**  
    Ожидаемый ответ: репозиторий, registry/CI, окружения, секреты, домены, бэкапы БД, админские аккаунты.

16. **Как подтверждается, что “передача завершена”?**  
    Ожидаемый ответ: чеклист приемки выполнен полностью (см. раздел 3), включая запуск с нуля и демонстрацию ключевых сценариев.

## 2) Публичные интерфейсы, которые нужно зафиксировать в handoff

- **Auth API**: `/auth/login`, `/auth/register`, `/auth/demo`, JWT claim’ы (`sub`, `role`).
- **Health-контракт**: `/health` как обязательный probe для окружений.
- **Лэндинг entry-router**: приоритет `admin_token`, fallback admin URL (`window.__DESKBOOK_ADMIN_URL` -> порт `5174`).
- **RBAC-контракт**: admin-only endpoint’ы (`/admin/*`, `/analytics` только для admin).

---

## 3) Live-checklist для встречи с техлидом

1. Открыть лэндинг в чистом браузере (без токенов), показать `User/Admin/Demo`.
2. Войти как пользователь, показать переход в user-app и базовое бронирование.
3. Войти как админ, показать переход в admin-app и admin-функции.
4. Проверить demo вход без регистрации и отсутствие админских операций.
5. Показать RBAC: user получает `403` на admin-only маршрут.
6. Показать конфликт бронирования: повторное пересечение дает `409`.
7. Прогнать smoke-тест целиком (`passed`).
8. Показать health endpoint’ы и логи контейнеров.

---

## 4) Assumptions для этой передачи

- Передача идет внешней команде (не внутреннему подразделению).
- Нужен материал для созвона/due-diligence, а не только список вопросов.
- Базовые ответы опираются на текущее состояние репозитория и реализованные `user/admin/demo` потоки.
- Орг-вопросы (SLA, ownership, migration ownership) фиксируются отдельно в финальном handoff-протоколе.
